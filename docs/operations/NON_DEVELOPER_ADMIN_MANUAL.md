# 비개발자 운영자용 Admin 매뉴얼

이 문서는 개발 지식이 없는 운영자가 For-Ai admin 화면을 열었을 때 매일 무엇을 확인하고, 어떤 기준으로 claim을 검증하며, 커뮤니티·신고·비즈니스 정정 요청을 어떻게 처리할지 설명합니다.

For-Ai의 운영 목표는 “그럴듯한 글”을 많이 만드는 것이 아니라, AI·검색엔진·사람이 같은 출처를 인용할 수 있는 claim-level 사실 레지스트리를 유지하는 것입니다. 모르면 채우지 말고 `확인 필요` / `Needs verification`, confidence `low`, status `needs_review`로 남깁니다.

## 1. 매일 할 일

1. `/admin`에서 대시보드 집계를 불러옵니다.
2. `Pending claim reviews`를 먼저 확인합니다.
   - 새 claim, 수정 요청, stale claim, source check 실패가 있으면 오늘 처리할 우선순위를 정합니다.
3. `New hallucination reports`를 확인합니다.
   - AI 오답 신고는 공개된 잘못된 인용을 줄이는 데 직접 연결되므로 일반 커뮤니티 글보다 우선합니다.
4. `Business verification requests` 또는 business correction 요청을 확인합니다.
   - 사업자가 제출했더라도 자동 verified 처리하지 않습니다.
5. `New topic suggestions`와 생성 후보를 확인합니다.
   - 후보는 “검토할 주제”일 뿐이며, 사실로 승격하지 않습니다.
6. `Stale claims`를 확인합니다.
   - `last_verified_at` 기준 오래된 verified claim은 실제 출처를 다시 열어 값이 유지되는지 확인합니다.
7. `Recent admin actions`를 훑어봅니다.
   - 어제 처리한 항목이 의도대로 기록되었는지, 민감 정보가 metadata에 들어가지 않았는지 확인합니다.
8. 처리하지 못한 항목은 status와 notes에 이유를 남깁니다.
   - 예: “공식 출처 없음”, “출처끼리 충돌”, “사업자 증빙 요청 필요”.

## 2. Claim 검증 방법

For-Ai의 기본 단위는 문서 전체가 아니라 하나의 claim입니다. 하나의 claim은 하나의 검증 가능한 사실이어야 합니다.

검증 절차:

1. claim이 무엇을 말하는지 한 문장으로 분리합니다.
   - 예: “이 장소는 토요일 10:00에 영업한다.”
   - 예: “이 상품의 환불 기간은 구매 후 30일이다.”
2. claim이 특정 entity와 document에 연결되어 있는지 확인합니다.
   - `entity_id`가 없으면 먼저 entity를 정리해야 합니다.
3. claim 값이 현재 화면의 출처로 직접 확인되는지 확인합니다.
   - 출처가 주제를 설명하는 것만으로는 부족합니다. claim 값 자체를 뒷받침해야 합니다.
4. 출처의 발행 주체, 날짜, 관할권, 적용 범위를 확인합니다.
5. claim_sources에 출처 URL, 제목, source_type, 필요한 citation note를 남깁니다.
6. 검증 행동을 verification_events에 기록합니다.
7. 기준을 만족할 때만 status를 `verified`로 바꾸고 `last_verified_at`을 실제 검토 시점으로 갱신합니다.

검증 중 확인할 질문:

- 이 claim은 누구에게 적용되는가?
- 어느 국가·도시·관할권에 적용되는가?
- 현재도 유효한가?
- 출처가 claim 값을 직접 말하는가?
- 출처가 공식·1차 출처인가, 아니면 보조 출처인가?
- AI가 만든 문장이나 요약을 그대로 믿고 있지는 않은가?

## 3. 출처 좋은/나쁜 예시

### 좋은 출처

- 공식 웹사이트의 요금표, 운영시간, 정책 페이지
- 정부·규제기관·공공기관의 고시, 안내문, 법령, 공개 데이터
- 회사·기관이 직접 발행한 약관, 환불 정책, 제품 사양, 공지사항
- 공식 PDF, 공식 보도자료, 공식 신청 페이지
- 공인 등록부, 교통 운영사 데이터, 학교 입학처 안내, 병원 공식 진료 안내
- 날짜와 적용 범위가 명확한 1차 문서

### 제한적으로 쓸 수 있는 출처

- 신뢰할 수 있는 언론 기사
- 업계 데이터베이스나 플랫폼 listing
- 아카이브된 공식 페이지
- 여러 독립 출처가 같은 값을 말하는 secondary source

이 경우에는 confidence를 무조건 `high`로 올리지 말고, 한계와 확인 범위를 notes에 남깁니다.

### 나쁜 출처

- AI 챗봇 답변, AI 요약, 자동 생성 글
- 출처 없는 블로그 글, 광고성 글, 복사된 콘텐츠
- 오래된 커뮤니티 댓글 하나
- 날짜·관할권·적용 대상이 불분명한 글
- “다른 장소도 보통 이렇다”는 추정
- 스크린샷만 있고 원문 URL이나 발행 주체가 없는 자료
- 사업자가 보낸 주장만 있고 공개 또는 검증 가능한 증빙이 없는 자료

나쁜 출처는 조사 단서로만 사용할 수 있으며, 단독으로 verified 처리하면 안 됩니다.

## 4. Verified 처리 기준

claim을 `verified`로 처리하려면 모두 만족해야 합니다.

- 사람 운영자가 직접 검토했습니다.
- claim_sources에 traceable source가 있습니다.
- 출처가 claim 값을 직접 뒷받침합니다.
- source_type과 citation note가 운영자가 나중에 다시 이해할 수 있을 정도로 기록되어 있습니다.
- 출처의 적용 범위와 claim의 entity·jurisdiction·language가 맞습니다.
- 최신성이 중요한 claim은 날짜 또는 재검증 필요 주기를 고려했습니다.
- 충돌하는 출처가 있으면 충돌을 해결했거나 `disputed`로 남겼습니다.
- AI 생성 후보, 사업자 제출, 커뮤니티 제보를 사람 검토 없이 그대로 승인하지 않았습니다.

Verified 처리하면 안 되는 경우:

- 출처가 없거나 막연합니다.
- 공식 출처와 다른 출처가 서로 충돌합니다.
- 사용자가 “그렇다”고 말했지만 증빙이 없습니다.
- AI가 제안한 값만 있습니다.
- claim이 너무 넓어서 하나의 사실로 검증할 수 없습니다.
- 의료·법률·금융·안전 claim인데 출처가 보조 출처뿐입니다.

## 5. High-risk 주의사항

다음 claim은 high-risk로 보고 더 보수적으로 처리합니다.

- 의료, 건강, 병원 서비스, 유전자·DNA 검사, 개인정보 처리
- 법률, 비자, 정부 수수료, 행정 처리 기간, 자격 요건
- 금융 수수료, 금리, 환불, 보험, 투자 관련 조건
- 안전, 접근성, 장애인 편의, 긴급 서비스, 시설 제한
- 미성년자, 교육 입학, 시험 일정, 민감한 신원 정보
- 가격, 영업시간, 일정처럼 자주 바뀌는 정보

High-risk 운영 원칙:

- 공식 또는 1차 출처를 우선합니다.
- 보조 출처만 있으면 보통 confidence를 `medium` 이하로 유지합니다.
- 출처가 조금이라도 불명확하면 `needs_review` 또는 `disputed`로 둡니다.
- For-Ai는 의료·법률·금융 조언을 제공하지 않습니다. 사실 claim과 조언을 분리합니다.
- 개인의 민감정보, 원본 IP, 개인 DNA 데이터는 저장하지 않습니다.

## 6. 커뮤니티 글 처리 기준

커뮤니티 글은 사실 검증의 출발점일 수 있지만, 그 자체가 verified evidence는 아닙니다.

처리 기준:

- 스팸, 광고, 욕설, 개인정보 노출, 명예훼손 위험이 있는 글은 공개 노출을 제한하거나 삭제합니다.
- 새로운 사실 claim이 포함되어 있으면 claim 후보로 분리합니다.
- claim 후보는 공식·1차 출처로 별도 검증합니다.
- 단순 의견, 경험담, 리뷰는 claim으로 승격하지 않습니다.
- 커뮤니티 글에 있는 연락처, 예약번호, 개인 이름 등 민감정보는 보존하지 않습니다.
- 여러 사용자가 같은 문제를 제보하면 우선순위를 올릴 수 있지만, verified 근거로 보지는 않습니다.

## 7. AI 오답 신고 처리 기준

AI 오답 신고는 “AI가 틀렸을 가능성이 있는 claim”을 찾는 신호입니다.

처리 절차:

1. 신고 내용에서 문제 claim을 분리합니다.
2. 해당 entity/document/claim이 이미 있는지 찾습니다.
3. 기존 claim이 있으면 현재 값, status, confidence, last_verified_at을 확인합니다.
4. 공식·1차 출처로 실제 값을 확인합니다.
5. AI가 틀렸고 For-Ai claim도 틀렸다면 claim을 수정하고 source와 verification_event를 기록합니다.
6. AI는 틀렸지만 For-Ai claim은 맞다면 신고 기록에 “registry value correct”라고 남깁니다.
7. 출처 충돌 또는 불확실성이 있으면 `disputed` 또는 `needs_review`로 처리합니다.
8. 신고자가 제공한 개인정보나 원본 IP는 저장하지 않습니다. contributor_hash만 유지합니다.

AI 오답 신고는 신고 문장 자체를 사실로 믿지 않습니다. 반드시 출처 확인 후 처리합니다.

## 8. Business correction 처리 기준

사업자나 기관의 정정 요청은 중요하지만, For-Ai의 사실 무결성보다 우선하지 않습니다.

처리 기준:

- 사업자가 제출한 내용은 “우선 검토할 주장”으로 취급합니다.
- verified business profile이 있더라도 claim은 출처 기반으로 검증합니다.
- 사업자 주장과 공식 공개 페이지가 다르면 이유를 확인하고 필요한 증빙을 요청합니다.
- 정정 요청이 광고성 문구, 과장 표현, 비교 우위 주장이라면 claim으로 승격하지 않습니다.
- sponsored 또는 business-claimed content는 명확히 표시되어야 합니다.
- 금전 관계, 광고, affiliate, API tier는 verified 여부에 영향을 주면 안 됩니다.
- 사업자의 비공개 자료를 사용할 때는 공개 인용 가능 여부와 민감정보 포함 여부를 확인합니다.

승인 가능한 business correction 예시:

- 공식 홈페이지에 공개된 영업시간 변경
- 공개 약관에 반영된 환불 정책 변경
- 정부·플랫폼 등록부에 반영된 주소나 인허가 정보
- 사업자가 제공했고 운영자가 확인 가능한 공식 문서

거절 또는 보류해야 하는 예시:

- “우리 서비스가 최고” 같은 평가성 문구
- 경쟁사 비방 또는 확인 불가능한 비교 주장
- 출처 없는 가격·혜택·할인 주장
- 개인정보가 포함된 증빙
- 공개 출처와 충돌하지만 설명이 없는 요청

## 9. 운영자가 기억할 한 문장

확실하지 않으면 verified가 아닙니다. For-Ai에서는 빈칸을 추측으로 채우지 않고, `확인 필요`로 남기는 것이 올바른 운영입니다.
